#047

Migrate Stripe webhook handler to async queue

User Story
ASthe billing platform team
I WANTwebhook reception to be fast and resilient under load
SO THATStripe-side retries and duplicate state transitions stop causing incidents
Background

Today, every Stripe webhook triggers a synchronous cascade through three downstream services in the request path. Any slow handler — database contention, third-party API hiccup, GC pause — drags the entire webhook over Stripe's 10-second timeout.

When that timeout fires, Stripe retries the same event. Our handlers, lacking proper idempotency at the time, would replay state transitions: duplicate audit log rows, double subscription updates, occasionally double notifications to customers. Three production incidents in Q3 trace back to this pattern, and it burns engineering hours during every Q-end close.

This story is the first step in decoupling reception from processing. The HTTP receiver becomes a thin signature-verify + enqueue, and downstream work moves into a worker pool that we can scale, retry, and observe independently.

Acceptance Criteria 2 / 5 verified
  • Webhook handler logic is testable in isolation without HTTP fixtures
  • Duplicate Stripe events with the same event-id are deduplicated within a 24h window
  • P99 webhook latency stays below 500ms measured over a 7-day window post-cutover
  • Failed events land in DLQ with full original payload and last error trace
  • Cutover gated by a per-event-type feature flag with documented rollback procedure

#047

Migrate Stripe webhook handler to async queue

Async Stripe Webhook Pipeline

Stripe sends webhook events synchronously and expects a 2xx response within ~10 seconds, otherwise it retries. Our current handler runs all downstream logic — DB writes, audit-log emission, notification dispatch — in the request path. This couples webhook latency to whatever the slowest handler does, and a single slow handler can cascade into Stripe-side retries and duplicate processing.

Problem

  • P99 webhook latency drifted to 8.4s during peak hours (CloudWatch, last 30d).
  • Three production incidents in Q3 traced to webhook timeouts → Stripe retries → duplicate state transitions.
  • Handler logic is tested with HTTP-level fixtures; impossible to test retry/backoff behavior independently.

Proposed Design

Decouple webhook reception from event processing. The HTTP receiver validates the Stripe signature, deduplicates by event ID, enqueues the raw payload to SQS, and returns 200 immediately. A separate worker fleet consumes the queue and runs the existing handler logic with retry and DLQ semantics.

// Receiver — synchronous, fast path only
async fn handle_webhook(req: HttpRequest) -> Response {
    let event = verify_signature(&req)?;
    if redis.set_nx(&event.id, ttl: 86400).await? {
        sqs.send(&event.raw_body).await?;
    }
    Response::ok()
}

Why SQS over Kafka or in-process queue

SQS gives us managed durability, native DLQ, and visibility timeouts for retry semantics with zero ops cost. Kafka would be overkill for ~50 events/sec peak. In-process channels lose messages on deploy.

The trade-off: SQS adds ~80ms enqueue latency to the webhook path. Net latency still drops from 8.4s to ~250ms because we skip all downstream work.

Out of Scope

  • Webhook signature key rotation (tracked in #052).
  • Migrating non-Stripe webhooks (Paddle, Recurly) — separate evaluation.
